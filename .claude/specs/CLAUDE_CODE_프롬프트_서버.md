# Claude Code 프롬프트 — 서버 백엔드

아래 내용을 그대로 Claude Code에 전달한다.
(같은 저장소 루트에 `00_공통토대.md`, `01_서버_기획서.md`를 함께 두고 시작할 것.)

---

## 역할

self-hosted 음악 스트리밍 서비스의 **백엔드 서버**를 처음부터 구현한다.
같은 디렉토리의 `00_공통토대.md`(데이터 모델·API 계약)와 `01_서버_기획서.md`(서버 범위)를 **정본 명세로 삼는다.**
명세와 이 프롬프트가 충돌하면 명세를 따르고, 모호하면 질문한다.

## 핵심 설계 (반드시 준수)

- **논리 트랙(Track) ↔ 물리 소스(Source) 분리.** 한 곡이 여러 재생 수단(로컬 오디오/로컬 비디오)을 가진다. "파일 = 곡"으로 짜지 말 것.
- 메타데이터는 base(ID3→MusicBrainz→Last.fm) 위에 **유저 override 레이어가 최종 우선**. 자동 소스가 유저 입력을 덮지 않는다.
- 소스 선택·오디오↔비디오 오프셋 규칙은 `00_공통토대.md` §4, §5를 그대로 구현.

## 기술 스택 (확정)

- **NestJS + Fastify 어댑터** (Express 아님)
- **SQLite** + **better-sqlite3 / Drizzle** (Prisma 아님), **WAL 모드**
- 검색은 **SQLite FTS5** 내장 사용 (외부 검색 엔진 금지)
- 미디어 처리는 **ffmpeg / ffprobe 서브프로세스**
- 잡 큐는 **인프로세스**(p-queue 등). **Redis/BullMQ 등 외부 컴포넌트 추가 금지** (원클릭 self-host 목표)
- 실시간은 WebSocket 또는 SSE
- **OpenAPI 스펙을 자동 생성**(NestJS Swagger). 클라이언트가 이걸로 타입/클라이언트를 생성한다.

## MVP(v1) 범위 — 로컬 파일만

1. 데이터 모델 + 마이그레이션 (공통 토대 §2)
2. 라이브러리: 루트 디렉토리 풀스캔 + 파일시스템 감시(추가/수정/삭제 즉시 반영) + 수동 재스캔
3. 트랙↔비디오 자동 매핑 (신뢰도 점수, 임계값 미만은 "확인 대기 큐"로) + 중복 감지
4. 메타데이터 파이프라인 (override 레이어, 일괄 편집, 커스텀 아티스트, 커버곡 수동 매핑). MusicBrainz는 1req/s 큐+캐시
5. 스트리밍: HTTP Range(seek) + **seek 가능한 온더플라이 트랜스코딩 + 세그먼트 캐싱** + 유저별 비트레이트 상한
6. ReplayGain 산출·저장(값만 제공, 적용은 클라)
7. 풀텍스트 검색(FTS5)
8. 인증(토큰 기반, 관리자 초대제) + 멀티유저(플레이리스트/즐겨찾기/재생기록/설정 분리)
9. 델타 동기화(`since` 커서, tombstone) + 실시간 이벤트 push
10. OpenAPI 생성
11. **업로드 수집**: 웹 대시보드에서 다중 파일 업로드 → 관리 라이브러리 루트에 써넣고 동일 스캔/메타데이터 파이프라인으로 합류. 이름 변경(override), 커스텀 아티스트 수동 등록, 아트워크 업로드/교체 엔드포인트 포함.
12. **웹 정적 서빙**: 번들된 웹 클라이언트(플레이어 + 관리자 대시보드)를 서버 자기 주소에서 서빙(SPA fallback). 컨테이너 하나로 끝나게.
13. **통계 & 모니터링**: 사용자 재생 통계(PlayHistory 집계, 기간 필터) + 관리자 운영 통계(활성 스트림·대역폭·트랜스코딩 부하·스토리지 분해). `GET /health`(Docker healthcheck), 선택적 `/metrics`(Prometheus). 트래픽 추적용 stream-session 카운터는 서버 내부에만 두고 sync 계약에 넣지 말 것.
14. **운영 유틸리티(admin)**: 라이브러리 무결성 점검(사라진 파일/고아 메타/깨진 아트워크), 트랜스코딩 캐시 비우기, 메타데이터·설정 백업/복원, 로그 조회.

**비범위(만들지 말 것)**: 외부 소스(YouTube 등), 스마트 플레이리스트, 관계도, 외부 가사, 공유 링크, 오프라인. 모델에 자리(`origin` 등)만 두되 구현은 하지 않는다.

## 반드시 지킬 안전장치

- **클라이언트 연결이 끊기면 해당 ffmpeg 프로세스를 즉시 종료**한다. 좀비 프로세스 누적 = 서버 다운. 트랜스코딩 캐시와 한 묶음으로 설계할 것.
- WAL 모드로 스캔 중 브라우징이 막히지 않게 한다.

## 권장 구현 순서

데이터 모델/마이그레이션 → 인증 → 스캔/감시 + 메타데이터 → 브라우징·검색 API → 스트리밍/트랜스코딩(가장 어려움, 일찍 검증) → 델타/실시간 → 매핑/일괄편집/관리자.

## 작업 규칙 (공통)

- **Docker 우선.** 원활한 유지보수를 위해 `Dockerfile` + `docker-compose.yml`을 처음부터 둔다. ffmpeg를 이미지에 포함하고, 라이브러리 경로·DB·캐시를 볼륨으로 분리. 로컬에서도 compose로 띄워 검증한다.
- **오픈소스 참고.** UX/동작 패턴(스캔 전략, 트랜스코딩 캐시, 스트리밍 동작 등)은 Navidrome, Jellyfin 같은 성숙한 self-host 프로젝트를 **참고**한다. 단 **코드를 복사하지 말 것**(AGPL 등 라이선스 문제) — 동작/설계 아이디어만 참고하고 구현은 직접 한다.
- **주기적 커밋.** 의미 있는 단위마다 자주 커밋한다. 한 번에 거대한 커밋을 만들지 않는다.
- **커밋 컨벤션**: Conventional Commits (`feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:` …)를 지킨다.
- **언어**: 커밋 메시지와 README는 **영어로** 작성한다.
- **코드 주석은 작성하지 않는다.** 코드는 이름과 구조로 의도를 드러낸다.
- **커밋에 본인 크레딧을 넣지 않는다.** `Co-authored-by`, "Generated with Claude Code" 등 AI 작성 표시를 커밋 메시지·푸터·README 어디에도 넣지 말 것.
- README에는 설정·실행(docker compose) 방법과 환경변수를 문서화한다.
