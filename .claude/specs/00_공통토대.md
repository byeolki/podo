# 공통 토대 — 데이터 모델 & API 계약 (v0.3)

## 0. 이 문서의 역할

서버 기획서와 클라이언트 기획서가 함께 참조하는 **단일 진실(single source of truth)**.
용어 정의, 데이터 모델, API 프로토콜의 큰 원칙만 담는다.
두 기획서가 세부를 정의하되, 데이터 모델·계약이 충돌하면 이 문서가 우선한다.

분량을 일부러 짧게 유지한다. 여기가 두꺼워지면 서버/클라 문서와 중복이 생긴다.

---

## 1. 핵심 설계 원칙 — 논리 트랙 ↔ 물리 소스 분리

이 프로젝트의 차별점(트랙별 소스 우선순위, 오디오↔비디오 전환, 커스텀 아티스트, 커버곡 매핑)은
전부 아래 한 가지 분리 위에서만 성립한다. Subsonic 계열이 따라오지 못하는 지점이자, 처음부터 다시 짜는 이유다.

- **Track(논리)** — 추상적인 "곡". 플레이리스트·큐·재생기록·즐겨찾기가 가리키는 단위. 재생 방법을 직접 들고 있지 않다.
- **Source(물리)** — 그 곡을 실제로 재생하는 수단. 로컬 오디오 / 로컬 비디오 / (로드맵) 외부 스트림. 한 Track이 여러 Source를 가진다.

"파일 = 곡"이 아니라 "곡은 추상, 파일/스트림은 곡에 매달린 여러 재생 수단"이다.

## 2. 엔티티 정의 (MVP 필드 기준)

아래는 v1에서 실제로 필요한 필드만 적었다. 로드맵 전용 엔티티는 모델 자리만 잡고 표시한다.

### Track (논리적 곡)
- `id` — 서버 생성 불투명 식별자
- `title`
- `artist_ids` — 순서 있는 배열 (주 아티스트 + 피처링)
- `album_version_id` — nullable (싱글/미수록곡 가능)
- `track_number`, `disc_number`
- `canonical_duration` — 기준 타임라인 길이 (소스별 실제 길이와 다를 수 있음)
- `added_at`
- 해석된 메타데이터 = base 파싱 + 유저 override (§3)

### Source (물리적 재생 수단)
- `id`
- `track_id`
- `media_kind` — `audio` | `video`
- `origin` — `local` | (로드맵) `youtube` | `external`
- `format`, `codec`, `bitrate`, `sample_rate`
- `duration` — 이 소스 자체의 실제 길이
- `time_offset` — 트랙 기준 타임라인 정렬값 (기본 0, 오디오↔비디오 동기화용, §5)
- `priority` — 트랙 내 우선순위 (작을수록 우선)
- `locator` — 로컬 경로 또는 (로드맵) 외부 식별자
- `replaygain_track`, `replaygain_album` — nullable
- `available` — 현재 재생 가능 여부

### Artist
- `id` — 내부 고유 식별자 (외부 ID와 독립)
- `name`
- `is_custom` — Last.fm/MusicBrainz에 없는 자체 등록 여부
- `external_ids` — `{ musicbrainz, lastfm }`, nullable
- `created_by` — 커스텀 등록 시 유저

### Album / AlbumVersion
- **Album**: `id`, `title`, `primary_artist_id`
- **AlbumVersion**: `id`, `album_id`, `version_type`(`regular`|`repackage`|`remaster`|`single`|…), `release_year`, `artwork`
- Track은 하나의 AlbumVersion에 속한다. 하나의 Album 아래 여러 Version을 묶어 한 페이지에서 보여준다.

### User
- `id`, `name`, `role`(`admin`|`user`)
- `settings` — `{ max_bitrate_wifi, max_bitrate_cellular, … }`

### Playlist / Favorite / PlayHistory
- **Playlist**: `id`, `owner_user_id`, `name`, 순서 있는 `track_ids`
- **Favorite**: `user_id`, `track_id`
- **PlayHistory**: `user_id`, `track_id`, `source_id`, `played_at`, `played_duration`

### Lyrics
- `track_id`, `type`(`plain`|`synced`), `content`, `source`(`local`|`user`)
- 외부 가사 연동(Musixmatch 등)은 로드맵

### Tag
- `id`, `name`, `kind`(`genre`|(로드맵)`mood`) + Track↔Tag 조인
- 장르는 v1, 무드 태그 브라우징은 로드맵

### (로드맵) ArtistRelation
- 아티스트 간 엣지: `related` | `unit` | `collab`. 관계도 시각화와 함께 v2.

### TrackMetadataOverride
- 유저가 편집한 필드만 별도 저장. 해석 시 base 위에 최종 적용 (§3).

## 3. 메타데이터 해석 규칙

base 채움 순서는 빈 필드를 채워가는 방식이다:

1. 로컬 파일 ID3 태그
2. MusicBrainz
3. Last.fm

그 위에 **유저 override가 항상 최종 우선**한다.
즉 자동 소스는 유저가 입력/수정한 값을 절대 덮지 않는다.
(이전 기획서의 "유저 입력이 우선순위 꼴찌"는 차별점과 모순되어 폐기. 유저 입력은 우선순위가 아니라 override 레이어다.)

해석된 Track 메타데이터 = base 병합 결과에 override를 덮어쓴 뷰.

---

## 4. 소스 선택(resolution) 규칙

재생 요청 입력: `track_id` + 선택적 `media_kind` + 선택적 `source_id`(강제 지정).

서버 선택 절차:
1. `source_id`가 지정되면 그 소스 사용 (강제).
2. 아니면 `media_kind`로 필터 (미지정 시 `audio` 기본).
3. `available = true` 필터.
4. `priority` 오름차순 정렬 후 첫 소스.
5. 첫 소스가 재생 불가로 판명되면 다음 priority로 폴백.

## 5. 오디오 ↔ 비디오 정렬 규칙

같은 Track의 audio source와 video source는 타임라인이 어긋날 수 있다 (뮤직비디오 인트로/아웃트로 등).
단순 `currentTime` 복사로는 위치가 틀어진다.

정렬은 `time_offset`으로 한다:

```
video_position = audio_position + (video.time_offset - audio.time_offset)
```

v1은 소스당 단일 선형 오프셋만 지원한다.
구간별 정밀 매핑(여러 구간을 다른 비율로 맞추기)은 로드맵.

## 6. API 프로토콜 원칙 — 완전 독자 API

기존 Subsonic/OpenSubsonic 호환은 포기한다. 차별점을 담을 자리가 없기 때문이다.

- **형식**: REST + JSON over HTTPS. 베이스 경로 `/api/v1`.
- **인증**: 로그인 → 액세스 토큰(Bearer) + 리프레시 토큰. 모든 요청에 `Authorization` 헤더.
- **페이지네이션**: cursor 기반.
- **델타 동기화**: `?since=<cursor>`로 변경분만 수신. 엔티티별 변경 시각 + 삭제용 tombstone 제공.
- **실시간**: WebSocket(또는 SSE) 채널로 서버 → 클라 이벤트 push (`track.upserted`, `track.removed`, `scan.progress` 등).
- **스트리밍**: 별도 엔드포인트. HTTP Range 필수(seek). 쿼리로 트랜스코딩 타깃(format/bitrate)·소스 지정.
- **에러**: 표준 HTTP 코드 + JSON 바디(`code`, `message`).
- **버저닝**: URL prefix(`/api/v1`).

엔드포인트 상세 목록은 서버 기획서 §10에 둔다.

## 7. 표기 약속

- 필드명: `snake_case`
- 시각: ISO 8601, UTC
- 식별자: 서버 생성 불투명 문자열 (클라가 의미 추론 금지)
- 길이/오프셋 단위: 밀리초

---

## 8. 참조 관계

- 서버 기획서(`01`)는 이 모델의 **저장·해석 주체**를 정의한다.
- 클라이언트 기획서(`02`)는 이 모델을 **소비·표시하는 주체**를 정의한다.
- 두 문서의 "책임 경계" 절은 이 모델의 엔티티/규칙을 누가 소유하는지로 정해진다.
