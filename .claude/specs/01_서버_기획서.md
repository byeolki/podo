# 서버 기획서 — Self-hosted 음악 스트리밍 서버 (MVP / v1)

> 데이터 모델·용어·API 원칙은 **공통 토대 문서(`00`)** 를 따른다. 충돌 시 그쪽이 우선.

---

## 1. 목적 & 범위

AGPL-3.0 자체호스팅 서버. 음악의 저장·라이브러리·메타데이터·트랜스코딩·스트리밍·검색·인증·동기화를 담당하고,
클라이언트와는 **완전 독자 API(v1)** 로 통신한다. 서버는 라이브러리 상태의 단일 진실이다.

**v1은 로컬 파일만 다룬다.** 외부 소스(YouTube 등)는 데이터 모델에 자리(`origin`)만 두고 구현은 로드맵으로 미룬다.
(외부 소스 로직은 깨지기 쉬워 코어 안정성에 묶지 않는다.)

---

## 1-A. 기술 스택 (확정)

- **런타임/프레임워크**: Node.js(또는 Bun) + **NestJS**, HTTP 어댑터는 **Fastify** (Express 아님 — 스트리밍 처리량·백프레셔 유리)
- **DB**: **SQLite** (단일 인스턴스 self-host라 별도 DB 컨테이너 불필요). 접근은 **better-sqlite3 / Drizzle** (Prisma 아님 — FTS5 직접 제어 위해). **WAL 모드**로 스캔 중에도 브라우징이 막히지 않게.
- **검색**: **SQLite FTS5** 내장 (별도 검색 엔진 없음 → §7)
- **미디어**: **ffmpeg / ffprobe** 서브프로세스. 무거운 CPU 작업은 전부 ffmpeg로 떠넘겨 Node 단일 스레드 한계를 피한다.
- **잡 큐**: **인프로세스 큐**(p-queue 등). Redis/BullMQ는 쓰지 않는다 (원클릭 self-host 목표에 외부 컴포넌트 추가는 역행). 스캔·메타데이터 페치·MusicBrainz 1req/s 제한을 여기서 처리.
- **실시간**: WebSocket 또는 SSE (NestJS Gateway)
- **API 계약 공유**: NestJS Swagger 모듈로 **OpenAPI 자동 생성** → 웹은 TS 타입 공유, 네이티브는 OpenAPI에서 클라이언트 생성.

### 반드시 지킬 운영 안전장치
- **클라이언트 연결이 끊기면 해당 ffmpeg 프로세스를 즉시 종료한다.** (안 하면 좀비 프로세스 누적 → 서버 다운. 이 앱에서 가장 흔한 운영 사고이며 트랜스코딩 캐시 설계와 한 묶음 — §6.3)

## 2. 책임 경계 — 서버가 가지는 것

서버가 소유: 파일 저장·스캔·감시, 메타데이터 해석, 소스 선택, 트랜스코딩, ReplayGain 산출, 스트리밍,
검색 인덱스, 인증·계정, 델타/실시간 동기화, 라이브러리 상태의 단일 진실.

서버가 가지지 않는 것(클라 기획서 소관): 재생 엔진, 크로스페이드·이퀄라이저 같은 클라측 DSP, UI, OS 네이티브 통합.
경계 원칙은 "디코딩된 오디오가 로컬에 있어야 하는 처리(크로스페이드/EQ)는 클라, 그 외 라이브러리·소스 처리는 서버".

## 4. 라이브러리 관리

### 4.1 스캔 & 감시
- 관리자가 루트 디렉토리를 등록한다.
- 등록 시 초기 풀스캔, 이후 파일시스템 이벤트로 추가/수정/삭제를 즉시 반영한다.
- 수동 재스캔 트리거 제공.
- 스캔 진행상태는 실시간 채널로 방송한다 (`scan.progress`).

### 4.2 지원 포맷
- 오디오: mp3, m4a, flac, aac, wav, ogg, opus
- 비디오: mp4, m4v, mkv

### 4.3 트랙 ↔ 비디오 매핑
- **자동 매핑**: 정규화된 (아티스트, 제목)을 기준으로 audio source와 video source를 같은 Track에 연결한다.
- 자동 매칭에는 **신뢰도 점수**를 붙인다.
- 신뢰도가 임계값 미만이면 자동 연결하지 않고 **확인 대기 큐**로 보낸다. 관리자가 승인/거부/수동 연결.
  (단순 "자동 연결"은 동명이곡·라이브 버전 등에서 오연결을 낸다.)

### 4.4 중복 감지
- 동일 곡의 중복 소스 후보를 감지한다 (태그 + 길이/오디오 지문 기준).
- 관리자에게 알리고, 하나의 Track으로 병합할지 별개로 유지할지 선택하게 한다.

### 4.5 앨범 버전
- 정규/리패키지/리마스터/싱글을 AlbumVersion으로 구분하고, 하나의 Album 아래 묶는다.

### 4.6 업로드 수집 (웹 대시보드 경유)
수집 경로는 두 가지이며 **같은 스캔/메타데이터 파이프라인으로 합류**한다:
- (A) 기존: 관리자가 채운 디렉토리를 서버가 감시
- (B) 신규: **웹 대시보드에서 파일 업로드** → 서버가 관리 라이브러리 루트에 써넣음 → 동일 파이프라인 통과

세부:
- 다중 파일 업로드 + 진행상태. 지원 포맷 검증(§4.2). 용량/디스크 가드.
- 업로드 직후 메타데이터를 인라인으로 보정(제목/아티스트/앨범)한 뒤 커밋 가능.
- 업로드된 파일도 중복 감지·트랙↔비디오 매핑 대상.
- 권한: 관리자(또는 허용된 유저)만.

## 5. 메타데이터 파이프라인

- base 채움 순서: ID3 → MusicBrainz → Last.fm (빈 필드만 다음 소스가 채움).
- MusicBrainz는 초당 1요청 제한이 있어 **요청 큐 + 캐시**로 처리한다 (대량 스캔 시 병목 방지).
- **유저 override가 최종 우선** (공통 토대 §3).
- **태그 일괄 편집**: 여러 Track의 override를 한 번에 적용. 옵션으로 로컬 파일 ID3에 write-back.
- **커스텀 아티스트/앨범**: `is_custom`으로 외부 ID 없이 생성(외부 조회 없이 수동 입력으로 충분). 후에 외부 매칭이 생기면 승격/병합.
- **이름 변경**: 트랙/앨범/아티스트 이름 변경은 override 적용으로 처리(별도 동작이 아니라 같은 레이어).
- **아트워크**: 앨범/트랙 커버 이미지 업로드·교체 지원.
- **커버곡 매핑**: 로컬 source를 원곡 Track/Artist에 수동 연결. 원곡 아티스트 페이지에서 커버도 함께 노출.

## 6. 소스 & 스트리밍

### 6.1 소스 선택
공통 토대 §4 규칙을 구현한다.

### 6.2 스트리밍 엔드포인트
- HTTP Range 지원으로 seek 가능.
- 원본 통과(무손실 FLAC) 또는 온더플라이 트랜스코딩.
- 유저별 max bitrate 상한을 적용. Wi-Fi/셀룰러 구분은 클라가 파라미터로 요청하고, 서버는 유저 상한을 캡으로 건다.

### 6.3 트랜스코딩 — 난이도 높음 (명시)
- ffmpeg 기반. FLAC → AAC / Opus.
- **seek 가능한 온더플라이 트랜스코딩**이 서버 정확성에서 가장 어렵다.
  range 요청 위치에서 트랜스코딩을 시작하고 세그먼트를 캐싱하는 방식이 필요하다.
- 이 부분은 별도 기술 설계에서 가장 먼저 풀 항목으로 잡는다.

### 6.4 ReplayGain
- 스캔 시 track/album gain을 산출·저장한다.
- 값만 스트림 메타로 제공하고, 실제 볼륨 적용은 클라가 한다.

## 7. 검색 (이전 기획서 누락분 보강)

> 이전 문서에는 관계도·태그·스마트 플레이리스트·통계는 있었으나 **일반 텍스트 검색이 없었다.**
> 라이브러리가 커질수록 가장 많이 쓰는 기능이라 v1 코어로 포함한다.

- 서버 풀텍스트 인덱스: 트랙/아티스트/앨범 대상.
- 부분일치 + 정규화(대소문자·발음기호). 한글 초성 검색은 nice-to-have(로드맵 후보).
- 결과를 타입별로 그룹화, cursor 페이지네이션.

## 7-A. 통계 & 모니터링

### 재생 통계 (사용자)
- PlayHistory 기반 집계. 최다 재생 트랙/아티스트/앨범, 기간 필터(주간/월간/전체), 총 청취 시간, 최근 추가·최근 재생.

### 운영·트래픽 통계 (관리자)
- **활성 스트림**: 현재 누가 무엇을 재생 중인지 실시간.
- **대역폭 사용량**: 유저별 / 전체, 기간별.
- **트랜스코딩 부하**: 동시 작업 수, 캐시 적중률.
- **요청량**, **스토리지 분해**(라이브러리 / 트랜스코딩 캐시 / 업로드·아트워크), **스캔 이력**.
- 경량 추적용 stream-session / usage 카운터를 서버 내부에 둔다(델타 sync 계약에는 포함하지 않음 — 클라가 동기화할 데이터가 아님).

### 헬스 / 메트릭
- `GET /health` — Docker healthcheck용.
- `GET /metrics` — (선택) Prometheus 포맷. Grafana 연동 가능.

### 프라이버시
- 멀티유저에서 개별 청취 내역은 유저 본인 소유다. 관리자에겐 집계·운영 지표 위주로 노출하고, 개별 청취 노출 여부는 설정 토글로 둔다.

---

## 7-B. 운영 유틸리티 (관리자)

- **라이브러리 무결성 점검**: DB에는 있으나 파일이 사라진 항목, 고아 메타데이터, 깨진 아트워크를 탐지·정리.
- **트랜스코딩 캐시 관리**: 용량 확인 및 비우기(캐시는 방치 시 비대해짐).
- **백업 / 복원**: 메타데이터 DB·설정 내보내기/가져오기 (데이터 주권 — 기획 핵심 가치).
- **로그 뷰어**: 최근 서버 로그 조회.
- (선택) 외부 노출 대비 요청 rate limiting.

---

## 8. 인증 & 멀티유저

- **관리자 초대제**. 토큰 기반 인증(access + refresh).
- 유저별 분리: 플레이리스트 / 즐겨찾기 / 재생기록 / 스트리밍 비트레이트 설정.
- 관리자 권한: 유저 초대·관리, 라이브러리·스캔 관리, 매핑 확인 큐 처리.
- self-hosted 특성상 외부 노출 시나리오를 전제로 토큰 만료·갱신을 설계한다.

---

## 9. 동기화 & 실시간

- **델타 동기화**: 클라가 `since` 커서로 변경분만 받는다 (upsert + 삭제 tombstone). 라이브러리가 커도 전체 재수신 불필요.
- **실시간 채널**: 라이브러리 변경/스캔 진행 이벤트를 push 한다. 디렉토리 감시로 잡힌 변화가 클라까지 즉시 도달한다.

## 10. API 표면 (v1, 요약)

상세 스키마는 별도 API 명세서로 분리하되, v1 엔드포인트 윤곽은 다음과 같다.

**인증**
- `POST /api/v1/auth/login` — 토큰 발급
- `POST /api/v1/auth/refresh` — 토큰 갱신

**라이브러리 (브라우징)**
- `GET /api/v1/artists`, `GET /api/v1/artists/{id}`
- `GET /api/v1/albums`, `GET /api/v1/albums/{id}`
- `GET /api/v1/tracks`, `GET /api/v1/tracks/{id}` — Track + 소속 Source 목록 포함
- `GET /api/v1/sync?since=<cursor>` — 델타 동기화

**검색**
- `GET /api/v1/search?q=...&type=...`

**스트리밍**
- `GET /api/v1/stream/{track_id}?media_kind=&source_id=&format=&bitrate=` — Range 지원
- `GET /api/v1/artwork/{id}`, `GET /api/v1/lyrics/{track_id}`

**재생 상태**
- 큐/재생기록/즐겨찾기 동기화 엔드포인트 (`GET/PUT`)

**플레이리스트**
- `GET/POST/PATCH/DELETE /api/v1/playlists...`

**메타데이터 편집**
- `PATCH /api/v1/tracks/{id}/metadata` — override 적용 (이름 변경 포함)
- `POST /api/v1/tracks/bulk-metadata` — 일괄 편집
- `POST /api/v1/artists` — 커스텀 아티스트 생성(수동 입력)
- `POST /api/v1/tracks/{id}/mappings` — 커버곡/비디오 수동 매핑
- `PUT /api/v1/albums/{id}/artwork`, `PUT /api/v1/tracks/{id}/artwork` — 아트워크 업로드/교체

**업로드 & 관리자**
- `POST /api/v1/upload` — 파일 업로드(다중, 진행상태) → 관리 라이브러리 루트로 수집
- 스캔 트리거/상태, 유저 관리, 매핑 확인 큐(승인/거부/수동연결), 중복 병합
- 권한은 role(`admin`)로 게이팅

**통계 & 모니터링**
- `GET /api/v1/stats/me?period=` — 사용자 재생 통계
- `GET /api/v1/admin/stats/traffic?period=` — 대역폭/요청/트랜스코딩 부하
- `GET /api/v1/admin/streams` — 활성 스트림(실시간)
- `GET /api/v1/admin/storage` — 스토리지 사용량 분해
- `GET /health`, `GET /metrics`(선택)

**운영 유틸리티 (admin)**
- `POST /api/v1/admin/library/verify` — 무결성 점검
- `DELETE /api/v1/admin/cache/transcode` — 트랜스코딩 캐시 비우기
- `GET /api/v1/admin/backup`, `POST /api/v1/admin/restore` — 백업/복원
- `GET /api/v1/admin/logs` — 로그 조회

**웹 정적 서빙**
- `GET /` 이하 — 번들된 웹 클라이언트(플레이어 + 관리자 대시보드)를 서버 자기 주소에서 서빙 (§11)

**실시간**
- `GET /api/v1/events` (WebSocket/SSE)

## 11. 배포 & 운영

- **Docker Compose 원클릭** 실행.
- **서버가 웹 클라이언트(정적 빌드)를 자기 주소에서 서빙**한다. 컨테이너 하나로 플레이어 + 관리자 대시보드까지 끝 (Navidrome 방식). 네이티브 앱은 별도 배포.
- 설정은 환경변수로: 라이브러리 경로, DB, 업로드 저장 경로, (선택) 외부 API 키.
- 영속 볼륨: 메타데이터 DB, 트랜스코딩 캐시, 검색 인덱스, 업로드/아트워크 저장소.
- 외부 노출 시 리버스 프록시 + HTTPS 권장.
- compose에 `/health` 기반 Docker healthcheck를 건다.
- 라이선스: **AGPL-3.0** (서버에만 적용. 클라이언트는 네트워크 API로 통신하는 독립 프로그램이라 전염되지 않음).
